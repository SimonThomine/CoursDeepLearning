Normalizaci√≥n por lotes (Batch Normalization)#

La Normalizaci√≥n por Lotes (Batch Normalization) fue introducida en 2015 en el art√≠culo Batch Normalization: Accelerating Deep Network Training by Reducing Internal Covariate Shift. Tuvo un impacto significativo en el campo del Deep Learning. Actualmente, la normalizaci√≥n se utiliza casi sistem√°ticamente, ya sea BatchNorm, LayerNorm o GroupNorm (entre otras).

La idea detr√°s de BatchNorm es simple y est√° relacionada con el cuaderno anterior. Buscamos obtener preactivaciones que sigan una distribuci√≥n gaussiana en cada capa de la red. Hemos visto que una buena inicializaci√≥n permite lograr este comportamiento, pero no siempre es evidente, especialmente con muchas capas diferentes.

BatchNorm normaliza las preactivaciones con respecto a la dimensi√≥n del batch antes de pasarlas por las funciones de activaci√≥n. Esto garantiza una distribuci√≥n gaussiana en cada paso.

Esta normalizaci√≥n no afecta la optimizaci√≥n, ya que se trata de una funci√≥n derivable.

Reutilizaci√≥n del c√≥digo#

Vamos a reutilizar el c√≥digo del cuaderno anterior para implementar la normalizaci√≥n por lotes.

Implementaci√≥n de BatchNorm#

Seg√∫n el art√≠culo, esta es la informaci√≥n:

En primer lugar, se trata de normalizar.

Para ello, calculamos la media y la desviaci√≥n est√°ndar de hpreact y luego normalizamos con estos valores:

epsilon=1e-6
hpreact_mean = hpreact.mean(dim=0, keepdim=True)
hpreact_std= hpreact.std(dim=0, keepdim=True)
hpreact_norm = (hpreact - hpreact_mean) / (hpreact_std+epsilon)

Ahora podemos integrar esta normalizaci√≥n en la propagaci√≥n hacia adelante.

Antes de eso, observemos que a√∫n no hemos implementado la parte de escala y desplazamiento (scale and shift):

¬øPara qu√© sirve? La normalizaci√≥n confina los pesos a tomar valores de una gaussiana centrada y reducida. Esto limita la capacidad de expresi√≥n del modelo. Los par√°metros aprendibles \(\gamma\) y \(\beta\) permiten evitar este problema al a√±adir un desplazamiento con \(\beta\) y una escala con \(\gamma\).

Dado que se trata de par√°metros aprendibles, tambi√©n debemos agregarlos a los par√°metros del modelo:

C = torch.randn((46, embed_dim))
W1 = torch.randn((block_size*embed_dim, hidden_dim))*0.01 # On initialise les poids √† une petite valeur
b1 = torch.randn(hidden_dim) *0 # On initialise les biais √† 0
W2 = torch.randn((hidden_dim, 46))*0.01
b2 = torch.randn(46)*0 
# Param√®tres de batch normalization
bngain = torch.ones((1, hidden_dim))
bnbias = torch.zeros((1, hidden_dim))

parameters = [C, W1, b1, W2, b2, bngain, bnbias]
for p in parameters:
  p.requires_grad = True

Por lo tanto, en la propagaci√≥n hacia adelante tendremos:

batch_size = 32

ix = torch.randint(0, Xtr.shape[0], (batch_size,))
  
# Forward
Xb, Yb = Xtr[ix], Ytr[ix] 
emb = C[Xb] 
embcat = emb.view(emb.shape[0], -1)
hpreact = embcat @ W1 + b1 

# Batch normalization
bnmean = hpreact.mean(0, keepdim=True)
bnstd = hpreact.std(0, keepdim=True)
hpreact = bngain * (hpreact - bnmean) / bnstd + bnbias

h = torch.tanh(hpreact) 
logits = h @ W2 + b2 
loss = F.cross_entropy(logits, Yb)

El problema de la Normalizaci√≥n por Lotes#

Al reflexionar un poco, podemos identificar problemas potenciales relacionados con BatchNorm:

Un ejemplo se ve afectado por otros elementos del batch: Normalizar seg√∫n la dimensi√≥n del batch significa que los valores de cada ejemplo dentro del batch est√°n influenciados por los otros ejemplos. Esto podr√≠a parecer problem√°tico, pero en la pr√°ctica es algo positivo. El uso de batches aleatorios en cada √©poca permite una regularizaci√≥n, lo que reduce el riesgo de sobreajuste (overfit) en los datos. Sin embargo, si queremos evitar este problema, podemos utilizar otros m√©todos de normalizaci√≥n que no normalicen seg√∫n la dimensi√≥n del batch. En la pr√°ctica, la BatchNorm sigue siendo ampliamente utilizada porque funciona muy bien emp√≠ricamente.

Fase de prueba en un solo elemento: Durante el entrenamiento, cada elemento est√° influenciado por los otros elementos de su batch. Sin embargo, en la fase de inferencia, cuando usamos el modelo en un solo elemento, ya no podemos aplicar BatchNorm. Esto es un problema, ya que queremos evitar un comportamiento diferente durante el entrenamiento y la inferencia.

Para resolver este problema, tenemos dos soluciones:

  • Podemos calcular la media y la varianza sobre el conjunto de elementos al final del entrenamiento y usar estos valores. En la pr√°ctica, no queremos hacer una iteraci√≥n adicional sobre todo el conjunto de datos solo para esto, por lo que nadie lo hace as√≠.

  • Otra soluci√≥n consiste en actualizar la media y la varianza a lo largo del entrenamiento mediante un EMA (promedio m√≥vil exponencial). Al final del entrenamiento, tendremos una buena aproximaci√≥n de la media y la varianza de todos los elementos de entrenamiento.

En la pr√°ctica, podemos implementarlo en Python de la siguiente manera:

C = torch.randn((46, embed_dim))
W1 = torch.randn((block_size*embed_dim, hidden_dim))*0.01 # On initialise les poids √† une petite valeur
b1 = torch.randn(hidden_dim) *0 # On initialise les biais √† 0
W2 = torch.randn((hidden_dim, 46))*0.01
b2 = torch.randn(46)*0 
# Param√®tres de batch normalization
bngain = torch.ones((1, hidden_dim))
bnbias = torch.zeros((1, hidden_dim))
bnmean_running = torch.zeros((1, hidden_dim))
bnstd_running = torch.ones((1, hidden_dim))

parameters = [C, W1, b1, W2, b2, bngain, bnbias]
for p in parameters:
  p.requires_grad = True

batch_size = 32

ix = torch.randint(0, Xtr.shape[0], (batch_size,))
  
# Forward
Xb, Yb = Xtr[ix], Ytr[ix] 
emb = C[Xb] 
embcat = emb.view(emb.shape[0], -1)
hpreact = embcat @ W1 + b1 

# Batch normalization
bnmeani = hpreact.mean(0, keepdim=True)
bnstdi = hpreact.std(0, keepdim=True)
hpreact = bngain * (hpreact - bnmeani) / bnstdi + bnbias
with torch.no_grad(): # On ne veut pas calculer de gradient pour ces op√©rations
    bnmean_running = 0.999 * bnmean_running + 0.001 * bnmeani
    bnstd_running = 0.999 * bnstd_running + 0.001 * bnstdi

h = torch.tanh(hpreact) 
logits = h @ W2 + b2 
loss = F.cross_entropy(logits, Yb)

Nota: En nuestra implementaci√≥n, hemos elegido 0.001 para nuestro EMA. En la capa BatchNorm de PyTorch, este par√°metro est√° definido por momentum y su valor predeterminado es 0.1. En la pr√°ctica, la elecci√≥n de este valor depende del tama√±o del batch en relaci√≥n con el tama√±o del conjunto de datos de entrenamiento. Por ejemplo, para un batch grande con un conjunto de datos peque√±o, podemos tomar 0.1. Para un batch peque√±o con un conjunto de datos grande, es mejor elegir un valor m√°s peque√±o.

Probemos ahora el entrenamiento de nuestro modelo para verificar que la capa funcione. Para este peque√±o modelo, no habr√° diferencia en el rendimiento.

lossi = []
max_steps = 200000

for i in range(max_steps):
  ix = torch.randint(0, Xtr.shape[0], (batch_size,))
  Xb, Yb = Xtr[ix], Ytr[ix] 
  emb = C[Xb] 
  embcat = emb.view(emb.shape[0], -1)
  hpreact = embcat @ W1 + b1 
  
  # Batch normalization
  bnmeani = hpreact.mean(0, keepdim=True)
  bnstdi = hpreact.std(0, keepdim=True)
  hpreact = bngain * (hpreact - bnmeani) / bnstdi + bnbias
  with torch.no_grad(): # On ne veut pas calculer de gradient pour ces op√©rations
      bnmean_running = 0.999 * bnmean_running + 0.001 * bnmeani
      bnstd_running = 0.999 * bnstd_running + 0.001 * bnstdi
    
  h = torch.tanh(hpreact) 
  logits = h @ W2 + b2 
  loss = F.cross_entropy(logits, Yb)
  
  for p in parameters:
    p.grad = None
  loss.backward()
  lr = 0.1 if i < 100000 else 0.01 
  for p in parameters:
    p.data += -lr * p.grad
  if i % 10000 == 0:
    print(f'{i:7d}/{max_steps:7d}: {loss.item():.4f}')
  lossi.append(loss.log10().item())

Consideraciones adicionales#

Sesgo: La normalizaci√≥n por lotes normaliza las preactivaciones de los pesos. Esta normalizaci√≥n anula el sesgo (ya que este desplaza la distribuci√≥n, mientras que nosotros la recentramos). Cuando se utiliza BatchNorm, se puede prescindir del sesgo. En la pr√°ctica, si se deja un sesgo, no causa problemas, pero es un par√°metro de la red que ser√° in√∫til.

Ubicaci√≥n de BatchNorm: Seg√∫n lo que hemos visto, es l√≥gico colocar BatchNorm antes de la funci√≥n de activaci√≥n. En la pr√°ctica, algunos prefieren colocarla despu√©s de la capa de activaci√≥n, por lo que no se sorprenda si se encuentra con esto en la literatura o en alg√∫n c√≥digo.

Otras normalizaciones#

Vamos a hacer un r√°pido repaso de las otras normalizaciones utilizadas para el entrenamiento de redes neuronales.

Figura extra√≠da del art√≠culo.

Normalizaci√≥n por Capa (Layer Normalization): Esta capa de normalizaci√≥n tambi√©n se utiliza con mucha frecuencia, especialmente en modelos de lenguaje (GPT, Llama). Consiste en normalizar sobre el conjunto de activaciones de la capa en lugar de sobre la dimensi√≥n del batch. En nuestra implementaci√≥n, esto simplemente implicar√≠a cambiar:

# Batch normalization
bnmeani = hpreact.mean(0, keepdim=True)  
bnstdi = hpreact.std(0, keepdim=True)   
# Layer normalization
bnmeani = hpreact.mean(1, keepdim=True)  
bnstdi = hpreact.std(1, keepdim=True)

Normalizaci√≥n por Instancia (Instance Normalization): Esta capa normaliza las activaciones en cada canal de cada elemento de manera independiente.

Normalizaci√≥n por Grupo (Group Normalization): Esta capa es una especie de fusi√≥n entre LayerNorm e InstanceNorm, ya que se calcula la normalizaci√≥n sobre grupos de canales (si el tama√±o de un grupo es 1, es InstanceNorm, y si el tama√±o de un grupo es \(C\), es LayerNorm).
