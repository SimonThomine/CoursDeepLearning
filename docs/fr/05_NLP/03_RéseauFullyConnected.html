R√©seau fully connected#

Architecture#

Dans les cours pr√©c√©dents (cours 2), nous avons construits des r√©seaux de neurones fully connected pour des probl√®mes de classification. Ici, nous avons √† faire √† un probl√®me de pr√©diction et nos donn√©es sont discr√®tes.

Inspiration#

Le r√©seau construit dans ce notebook est bas√© sur l‚Äôarticle ‚ÄúA Neural Probabilistic Language Model‚Äù.

Voici √† quoi ressemble l‚Äôarchitecture de ce r√©seau :

Figure extraite de l‚Äôarticle original.

Dans l‚Äôarticle, le mod√®le utilise des mots en entr√©e (3 mots dans l‚Äôexemple illustr√©) pour pr√©dire le mot suivant. Dans notre cas, nous allons utiliser des caract√®res, comme nous l‚Äôavons fait dans le notebook pr√©c√©dent.

Matrice d‚Äôembedding \(C\) : √Ä premi√®re vue, on remarque que le r√©seau comporte une matrice \(C\) qui va encoder le mot (ou le caract√®re) dans un espace latent. C‚Äôest une pratique presque syst√©matiquement utilis√© en NLP car cela permet de rapprocher les mots ‚Äúsimilaires‚Äù dans un espace latent.
Par exemple, dans la plupart des phrases on pourrait interchanger les mots ‚Äúchien‚Äù et ‚Äúchat‚Äù, ce qui signifie que ces mots vont avoir une representation proche dans l‚Äôespace latent tandis que les mots ‚Äúchien‚Äù et ‚Äúest‚Äù ne vont pas avoir une representation proche.

Reste du r√©seau : Le reste du r√©seau est plus classique, il prend en entr√©e la concatenation des embeddings des diff√©rents mots (ou caract√®res) et pr√©dit un mot (ou caract√®re en sortie).

Le mod√®le de l‚Äôarticle est entra√Æn√© par minimisation du negative log likelihood (comme ce que nous avons fait dans le notebook pr√©c√©dent avec le mod√®le bigramme).

Notre approche#

Dans l‚Äôarticle, ils utilisent 3 mots pour pr√©dire le 4√®me mot. Nous allons nous baser sur le m√™me principe et pr√©dire le 4√®me caract√®re √† partir des 3 caract√®res pr√©c√©dents.
La dimension de l‚Äôespace latent utilis√© dans l‚Äôarticle est 30 pour un dictionnaire contenant 17 000 mots distincts. Comme nous avons 46 caracact√®res, prenons une dimension d‚Äôembedding de 10 assez arbitrairement.

Comment trouver le bon learning rate ?#

Le choix du learning rate est tr√®s important lors de l‚Äôentra√Ænement d‚Äôun r√©seau de neurones et il est souvent difficile de savoir quelle valeur choisir avant d‚Äôavoir fait des tests.
Une bonne fa√ßon de choisir le learning rate est la suivante :
On construit une liste de 1000 valeurs entre -3 et 0 puis on prend \(10^{valeur}\) pour chaque valeur Cela va nous donner une liste de valeurs entre \(10^{-3} = 0.001\) et \(10^{0}=1\) qui sont un panel de valeurs potentielles pour notre learning rate. Les valeurs -3 et 0 peuvent √™tre diff√©rentes, vous devez essayer de trouver des valeurs encadrant le learning rate optimal.

lre = torch.linspace(-3, 0, 1000)
lrs = 10**lre

On va ensuite tracker les valeurs de loss en fonction du learning rate sur l‚Äôensemble de nos valeurs sur l‚Äôentra√Ænement.

lri = []
lossi = []
count=0
while count<999:
  for x,y in train_loader:
    count+=1
    if count==999:
        break
    # forward pass
    emb = C[x]
    h = torch.tanh(emb.view(-1, block_size*embed_dim) @ W1 + b1)
    logits = h @ W2 + b2 
    loss = F.cross_entropy(logits, y)
    
    # retropropagation
    for p in parameters:
        p.grad = None
    loss.backward()
    
    # Mise √† jour des poids du mod√®le
    lr = lrs[count]
    for p in parameters:
        p.data += -lr * p.grad

    lri.append(lre[count])
    lossi.append(loss.log10().item())

Avec cette courbe, on peut d√©duire qu‚Äôune bonne valeur de learning rate se situe aux alentours de \(10^{-1}\) et \(10^{-0.5}\).
On va donc choisir un learning rate de 0.2 que l‚Äôon diminuera au cours de l‚Äôentra√Ænement (pratique courante pour une convergence rapide et un optimisation pr√©cise sur la fin de l‚Äôentra√Ænement).

Point sur la tangente hyperbolique#

Dans notre optimisation, nous avons utilis√© la fonction tangente hyperbolique comme fonction d‚Äôactivation.
Elle est d√©finie de la mani√®re suivante :
\(\tanh(x) = \frac{\sinh(x)}{\cosh(x)} = \frac{e^x - e^{-x}}{e^x + e^{-x}}\)
Et on peut la visualiser en python :

En g√©n√©ral, dans les couches cach√©es de notre r√©seau, on privilegie l‚Äôutilisaton de tanh plut√¥t que sigmoid et cela pour plusieurs raisons :

  • La plage de sortie centr√©e sur z√©ro (-1 √† 1) facilite l‚Äôapprentissage.

  • Les gradients sont plus importants pour des valeurs entre -2 et 2 que pour la fonction sigmo√Øde

  • Les deux √©lements pr√©c√©dents conduisent √† une diminution du probl√®me de vanishing gradient et permettent une convergence plus rapide lors de l‚Äôentra√Ænement.

Optimisation du r√©seau#

Passons maintenant √† l‚Äôoptimisation de notre r√©seau. D√©finissons nos hyperparam√®tres :

lr=0.2
epochs=100

# Reinitialisons les param√®tres pour plus de simplicit√© si on a besoin de relancer l'entra√Ænement
C = torch.randn((46, embed_dim))
W1 = torch.randn((block_size*embed_dim, hidden_dim))
b1 = torch.randn(hidden_dim)
W2 = torch.randn((hidden_dim, 46))

stepi = []

  loss_epoch=0
  for x,y in train_loader:
    
    # forward pass
    emb = C[x]
    h = torch.tanh(emb.view(-1, block_size*embed_dim) @ W1 + b1)

    loss.backward()
    
    # Mise √† jour des poids du mod√®le

  stepi.append(epoch)

  # Calcul du loss de validation (pour surveiller l'overfitting)

  for x,y in val_loader:
